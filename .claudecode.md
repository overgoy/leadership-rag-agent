# Project Rules & Best Practices for Claude Code

## 1. Tech Stack
- Python 3.11+, `litellm`, `tavily-python` (or `duckduckgo-search`), `pydantic`, `streamlit`, `sqlite3`, `python-dotenv`.

## 2. Architecture & Design Patterns
- **Paradigm:** Lean and maintainable. No overengineering.
- **Hybrid Structured RAG:** No Vector DBs. We use Structured Extraction to SQLite for exact entity matches (Text-to-SQL). For semantic/context queries, we utilize SQLite's native FTS5 (Full-Text Search) extension on a 'bio' column.
- **Target Scope:** Strictly extract only C-level executives, Vice Presidents, and Heads of departments (ignore junior staff or board members).
- **Indexing:** Ensure standard B-Tree indexes are created on heavily filtered columns (role, department) in SQLite.
- **Separation of Concerns:** Keep data collection logic independent from the UI and Database execution layers.
- **Data Provenance (Citations):** Extract and store the 'source_url' for each leader in SQLite. The chat agent MUST cite its sources when answering to ensure grounded generation.

## 3. Resilience & Performance
- **Validation:** Validate all LLM structured outputs using `pydantic`. Make non-critical fields `Optional` to prevent validation crashes.
- **Context Management:** Truncate raw web text to 15,000 chars to prevent token overflow.
- **Stability:** Use `litellm` built-in `num_retries=3` for API stability.
- **Concurrency:** Enable `PRAGMA journal_mode=WAL;` in SQLite for concurrent read/write performance.
- **UX & Memory:** Implement UI text streaming (`st.write_stream`) and strictly maintain conversation memory using Streamlit's `st.session_state`.
- **Caching:** Use basic caching (`@st.cache_data`) for DB schema reads to avoid redundant operations.

## 4. Security & Scalability
- **Security (Anti-Injection):** The chat agent (`app.py`) MUST connect to the SQLite database in strict **Read-Only (ro)** mode using URI connections (`file:...?mode=ro`) to prevent SQL injections or destructive actions (`DROP`, `DELETE`) generated by the LLM. 
- **Scalability:** Ensure the SQLite connection allows multithreading (`check_same_thread=False`) so the Streamlit UI can handle concurrent users.

## 5. Cost Optimization (FinOps)
- Use cost-effective "mini" models (e.g., `gpt-4o-mini` or `claude-3-haiku-20240307`) by default for extraction and Text-to-SQL. 
- Always define a sensible `max_tokens` limit (e.g., 1000) to prevent runaway generation costs.
- Prune the database schema injected into the LLM context (only include the necessary `leadership` table schema, no system tables).

## 6. Agentic Design (The Anthropic Way)
When building the Streamlit agent in `app.py`, strictly follow Anthropic's official Prompt Engineering best practices:
- **XML Tagging:** Structure all system prompts and injected contexts using XML tags (e.g., `<database_schema>`, `<user_query>`, `<instructions>`).
- **Forced Chain of Thought (CoT):** Instruct the LLM to wrap its reasoning process inside `<thinking>...</thinking>` tags before making a tool call or answering. Display this thinking process in the Streamlit UI using `st.expander` for transparency.
- **ReAct & Guardrails:** Use the system prompt to strictly bound the agent's persona. It MUST politely refuse off-topic questions.
- **Graceful Tool Failure:** If the `execute_sql` tool returns an error, pass the error back to the LLM in an `<error>` tag so it can self-correct its SQL syntax.
